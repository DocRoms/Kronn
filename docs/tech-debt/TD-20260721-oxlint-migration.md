- **Area**: Frontend / Tooling / CI
- **Problem (fact)**: Kronn currently runs ESLint 10 with typescript-eslint, React Hooks and React Refresh plugins. CI allows at most 162 warnings, and the flat config contains project-specific selectors, severities, file overrides and React export exceptions that a replacement must preserve. - **Why we can't fix now (constraint)**: Oxlint covers TypeScript and React ecosystems and provides ESLint migration tooling, but migration is not diagnostic-equivalent by construction. Kronn must prove parity for its strict TypeScript rules, React Hooks/Refresh warnings, `no-restricted-syntax` selectors and per-file relaxations before removing ESLint. In particular, Kronn budgets five separate React Compiler warnings (`purity`, `immutability`, `refs`, `set-state-in-effect`, and `preserve-manual-memoization`), while Oxlint exposes the React Compiler analysis as one experimental, off-by-default `react/react-compiler` rule; the current per-rule severities and counts therefore cannot be assumed equivalent. Oxlint's type-aware mode currently documents 59 of 61 rules, requires the TypeScript 7 native toolchain, may use substantial memory on very large codebases, and does not support some legacy `tsconfig` options. - **Impact**: A validated migration could shorten local and CI lint feedback, but an unchecked cutover could silently drop diagnostics or hide the existing warning debt. No Kronn-specific speedup has been measured yet. [src: url: https://oxc.rs/docs/guide/usage/linter.html]

- **Suggested direction (non-binding)**: Start with `@oxlint/migrate`, keep the generated config reviewed in-repo, and introduce Oxlint in a dual-run phase. Keep the TypeScript 7 compiler experiment independently reversible even if Oxlint type-aware checks later use the same native toolchain.
