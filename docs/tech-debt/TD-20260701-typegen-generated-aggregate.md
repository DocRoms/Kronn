- **ID**: TD-20260701-typegen-generated-aggregate
- **Area**: CI / Tooling / Frontend
- **Problem (fact)**: `make typegen` no longer regenerates `frontend/src/types/generated.ts`. It runs `cargo test export_bindings`, which (ts-rs 12) writes one file per type into `backend/bindings/` (gitignored, native `export type X = {...}` form, every `#[serde(default)]` field REQUIRED — no `?`). But the committed, frontend-consumed `generated.ts` is a single aggregate in `export interface X {...}` form with `#[serde(default)]` fields marked OPTIONAL (`?`). Nothing in the repo folds the bindings into the aggregate (checked: Makefile, `frontend/vite.config.ts`, `./kronn`, `entrypoint.sh`, git hooks, `.cargo/config`, package.json). The aggregate has drifted since ~2026-06-25. Proof ts-rs works: `backend/bindings/QuickPrompt.ts` correctly carries the new `agent_settings`.
- **Why we can't fix now (constraint)**: A straight concat of the bindings breaks ~189 frontend call sites — fields the app omits when constructing objects (e.g. `skill_ids`, `profile_ids`, `workspace_mode`) become required. A faithful generator must reproduce the serde-default→optional transform, and that information is NOT present in the ts-rs bindings (only in the Rust source `#[serde(default)]` / `skip_serializing_if` annotations). ts-rs 12 has no feature flag that auto-optionalizes serde-default fields (verified empirically: `skill_ids: Array<string>` emitted without `?`). Reproducing it needs either a Rust-source scan in the generator or hundreds of `#[ts(optional)]` annotations — a dedicated task, too risky to rush.
- **Impact**: dev friction (adding/changing a Rust model field silently fails to reach TS — the field must be hand-added to `generated.ts`), docs (the `make typegen` echo "Types written to generated.ts" is misleading).
- **Where (pointers)**: `Makefile` (`typegen:` target ~234), `backend/bindings/*.ts` (ts-rs output), `frontend/src/types/generated.ts` (committed aggregate), `backend/Cargo.toml:79` (ts-rs 12).
- **Suggested direction (non-binding)**: Add a `scripts/assemble-generated-types.mjs` wired into `make typegen` after the cargo step that: (1) reads `backend/bindings/*.ts`, strips per-file headers + `import` lines, `bigint`→`number`; (2) converts object `export type` → `export interface`; (3) marks a field optional (`?`) when the corresponding Rust field has `#[serde(default)]` / `skip_serializing_if` — parsed from `backend/src/models/*.rs`. Validate by running `tsc` over the whole frontend (the oracle). Alternatively migrate the frontend to tolerate required fields (bigger app-code churn). As an interim, new fields are hand-added to `generated.ts` matching the existing interface style (source of truth = the Rust model).
