- **ID**: TD-20260721-edit-resend-presence-dispatch
- **Area**: Backend / Frontend / Discussions / MCP presence
- **Problem (fact)**: The UI's “Edit and resend” path deletes trailing Agent/System rows, updates the last User row in place, then calls `/api/discussions/{id}/run`. That endpoint intentionally bypasses the live-MCP-participant guard used by the normal message-send path, so it starts a local CLI agent in addition to connected room peers. The edit itself keeps the message's existing `sort_order`, so a peer waiting with `since_sort_order` cannot observe the revised turn. This path also already has a frontend guard against rapid duplicate execution, showing that double-submit races are an established concern rather than a hypothetical edge case. - **Why we can't fix now (constraint)**: Reusing the normal presence guard without creating an observable edit event would suppress the local runner while connected peers remain asleep, producing no reply. Simply assigning `MAX(sort_order)+1` is also insufficient: deleting trailing rows lowers the maximum, so the replacement can reuse an order already covered by a peer cursor. The durable fix needs schema, API, frontend and federation changes and is separate from the 0.8.13 CI repair. - **Impact**: Editing a shared-room turn can spawn an unexpected paid/local agent, create duplicate responders, interrupt an existing orchestration session, and still fail to notify a persistent peer of the revised content. Reused ordering values can also make later replacement messages invisible to cursor-based waits.
- **Where (pointers)**: `frontend/src/pages/DiscussionsPage.tsx`, `frontend/src/lib/api.ts`, `backend/src/api/discussions/messaging.rs`, `backend/src/api/discussions/crud.rs`, `backend/src/db/discussions.rs`, `backend/src/api/federation.rs`, and the discussion-schema migrations.
- **Suggested direction (non-binding)**: Add a monotonic per-discussion event high-watermark column (for example `discussions.last_allocated_sort_order`), backfilled above the current maximum and incremented transactionally with `UPDATE ... RETURNING`; deletion must never reduce it. Every allocation path, including idempotent MCP bulk import, must use the same allocator. Replace the frontend's DELETE/PATCH/unguarded-run sequence with one atomic edit/resend endpoint that publishes a typed revision event with a fresh order, federates it when shared, and uses the presence lease/obligation CAS to choose exactly one responder owner: connected MCP peer or local runner. Specify the revision event's bridge representation and UI rendering (visible edited marker or intentionally hidden event); it must not masquerade as a new User message. Keep `/run` only as an explicit force-retry escape hatch, but require it to acquire a `runner_owned` obligation instead of bypassing dispatch ownership. Define whether revision history is retained or tombstoned before choosing between a new row and an in-place row update.
- **Acceptance criteria**: With one or more active MCP peers, an edited turn appears once as a typed revision beyond every peer's prior cursor and no local CLI starts; without a live peer, exactly one local runner starts; rapid double submit remains idempotent; deleting trailing replies never reuses a prior event order; a cursor greater than the maximum order of the surviving message rows is explicitly valid after deletion; MCP bulk import cannot bypass the shared allocator; a federated mirror receives the same revision semantics; explicit force retry starts exactly one local agent through a `runner_owned` obligation; tests cover concurrent edit/resend and the cursor high-watermark across deletion.
- **Next step**: write the schema/API contract and concurrency tests together with the pending discussion-presence lease/obligation work before implementing the UI switch.
